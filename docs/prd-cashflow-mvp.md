이 문서는 사용자가 제공한 PRD 초안을 저장소 기준 문서로 보관한 것이다.

| 프로젝트명 | 가칭: 캐시플로우 캘린더 / Cashflow Planner |

## 2. 서비스 개요

### 2.1 서비스 정의

본 서비스는 사용자의 통장, 신용카드, 체크카드, 고정수입, 고정지출, 일별 소비 내역을 기반으로 카드 결제일과 고정 지출일에 따른 미래 현금흐름을 예측하는 개인 재무관리 서비스이다.

일반 가계부가 과거 소비를 기록하는 데 초점을 둔다면, 본 서비스는 다음 질문에 답하는 것을 핵심 가치로 한다.

`다음 카드 결제일이 지나면 내 통장에 실제로 얼마가 남는가?`

### 2.2 핵심 문제 정의

사용자는 카드 사용액과 통장 잔고를 별도로 인지하는 경우가 많다.
특히 신용카드는 사용 시점과 실제 출금 시점이 다르기 때문에, 사용자는 다음과 같은 문제를 자주 겪는다.

| 문제 | 설명 |
|---|---|
| 카드 사용액과 현금 잔고의 괴리 | 신용카드로 소비했지만 통장 잔고는 아직 줄지 않아 실제 가용 현금을 착각함 |
| 카드 결제일 관리 어려움 | 카드별 결제일, 결제기간, 출금 통장이 달라 전체 흐름을 파악하기 어려움 |
| 고정지출 누락 | 월세, 보험료, 통신비, 구독료 등이 언제 어디서 빠지는지 놓치기 쉬움 |
| 월급 전후 현금 부족 | 카드 결제일과 수입일의 순서에 따라 일시적 현금 부족이 발생할 수 있음 |
| 통장별 역할 혼재 | 어떤 카드가 어떤 통장에 연결되어 있는지 관리하기 어려움 |

### 2.3 서비스 핵심 가치

- 통장과 카드의 연결 관계를 명확히 보여준다.
- 카드 결제일 기준으로 실제 출금 예정액을 계산한다.
- 고정수입과 고정지출을 반영하여 미래 잔고를 예측한다.
- 사용자가 오늘 지출한 금액이 다음 결제일에 어떤 영향을 주는지 보여준다.
- 사용자가 현재 소비 수준에서 앞으로 버틸 수 있는지를 판단하게 한다.

## 3. MVP 목표

### 3.1 MVP 한 줄 목표

사용자가 통장, 카드, 고정수입, 고정지출, 일별 지출을 입력하면 카드 결제일 기준의 미래 현금흐름을 확인할 수 있는 서비스 구현

### 3.2 MVP 검증 가설

| 구분 | 가설 |
|---|---|
| 가설 1 | 사용자는 자동 연동이 없어도 카드 결제 후 예상 잔고를 알기 위해 초기 데이터를 직접 입력할 의향이 있다. |
| 가설 2 | 사용자는 과거 소비 분석보다 다음 결제일 후 남는 돈에 더 큰 실질 가치를 느낀다. |
| 가설 3 | 카드 결제일, 월급일, 고정지출일을 한 화면에서 보면 현금 관리 스트레스가 줄어든다. |
| 가설 4 | 매일 지출 입력이 10초 이내라면 사용자는 수동 기록을 지속할 가능성이 있다. |

### 3.3 MVP 성공 기준

| 지표 | 목표 |
|---|---|
| 초기 설정 완료율 | 가입 후 통장, 카드, 수입, 지출 설정 완료율 50% 이상 |
| 지출 입력 지속률 | 7일 이상 지출 입력 유지 사용자 비율 30% 이상 |
| 대시보드 재방문율 | 초기 설정 후 3일 이내 대시보드 재방문율 60% 이상 |
| 핵심 가치 인지 | 사용자 인터뷰에서 다음 카드값 후 잔고를 알 수 있다는 반응 확인 |
| 계산 신뢰도 | 수동 검산 기준 주요 예측 결과 오차 없음 |

## 4. MVP 범위

### 4.1 MVP 포함 범위

| 기능 영역 | 포함 여부 | 설명 |
|---|---|---|
| 통장 등록 | 포함 | 은행명, 통장명, 현재 잔고, 기준일 입력 |
| 카드 등록 | 포함 | 신용카드, 체크카드 구분 |
| 카드 결제일 등록 | 포함 | 신용카드별 결제일 입력 |
| 카드 결제기간 등록 | 포함 | 사용자가 직접 결제기간 입력 |
| 카드-통장 연결 | 포함 | 카드 결제금액이 어느 통장에서 빠지는지 설정 |
| 고정수입 등록 | 포함 | 월급, 부업수입 등 반복 수입 등록 |
| 고정지출 등록 | 포함 | 월세, 보험료, 통신비, 구독료 등 등록 |
| 초기 잔고 입력 | 포함 | 서비스 시작 시점의 통장별 잔고 입력 |
| 초기 카드 미청구액 입력 | 포함 | 이미 사용했으나 아직 결제되지 않은 카드 금액 입력 |
| 일별 지출 입력 | 포함 | 날짜, 금액, 결제수단, 사용처, 카테고리 입력 |
| 카드 결제예정액 계산 | 포함 | 결제기간 기준 카드별 예정액 계산 |
| 현금흐름 대시보드 | 포함 | 현재 총 현금, 예정 수입/지출, 결제 후 예상 잔고 표시 |
| 엑셀 업로드 | 제한적 포함 | 정해진 템플릿 기반 초기 데이터 입력용으로만 제공 가능 |

### 4.2 MVP 제외 범위

| 기능 | 제외 사유 |
|---|---|
| 은행/카드 자동 연동 | 금융 인증, 보안, API 연동, 법적 검토 필요 |
| 카드사별 자동 결제기간 수집 | 카드사별 정책 차이로 초기 MVP 복잡도 증가 |
| 문자/알림 자동 파싱 | 예외 케이스가 많고 정확도 검증 필요 |
| 할부 결제 | 결제 회차, 청구월, 수수료 등 복잡도 높음 |
| 리볼빙/현금서비스 | 금융상품 성격이 강하고 정책 설계 필요 |
| 선결제 자동 반영 | MVP에서는 수동 조정으로 처리 |
| 투자자산 관리 | 현금흐름 예측이라는 MVP 목적과 다름 |
| 영수증 OCR | 필수 기능 아님 |
| AI 소비 분석 | 충분한 데이터 축적 이후 확장 |
| 가족/공동계좌 | 권한, 공유, 개인정보 이슈 존재 |

## 5. 주요 사용자 정의

### 5.1 핵심 타깃 사용자

| 사용자 유형 | 설명 |
|---|---|
| 월급 기반 생활자 | 매월 정해진 수입이 있고 카드 결제일 관리가 필요한 사용자 |
| 신용카드 다중 사용자 | 여러 개의 신용카드를 사용하며 결제일과 출금 통장이 다른 사용자 |
| 고정지출이 많은 사용자 | 월세, 보험료, 통신비, 구독료 등 반복 지출이 많은 사용자 |
| 월급 전 현금 부족 경험자 | 카드 사용액과 통장 잔고를 혼동해 월말 현금 부족을 경험한 사용자 |
| 수동 가계부 사용자 | 자동 연동보다 직접 입력과 통제감을 선호하는 사용자 |

### 5.2 주요 사용자 니즈

| 니즈 | 설명 |
|---|---|
| 다음 카드값 확인 | 이번 결제일에 얼마가 빠질지 알고 싶다 |
| 결제 후 잔고 확인 | 카드값이 빠진 뒤 실제 현금이 얼마나 남는지 알고 싶다 |
| 고정지출 관리 | 매월 언제 어떤 돈이 빠지는지 한눈에 보고 싶다 |
| 카드별 사용액 관리 | 카드별 현재 사용액과 결제예정액을 보고 싶다 |
| 통장별 출금 흐름 확인 | 어떤 통장에서 어떤 돈이 빠지는지 알고 싶다 |

## 6. 핵심 사용자 시나리오

### 6.1 신규 사용자 초기 설정 시나리오

1. 사용자가 서비스를 시작한다.
2. 통장 정보를 입력한다.
3. 보유한 신용카드와 체크카드를 입력한다.
4. 신용카드별 결제일과 결제기간을 입력한다.
5. 각 카드가 어느 통장에 연결되어 있는지 설정한다.
6. 월급 등 고정수입을 입력한다.
7. 월세, 보험료, 통신비, 구독료 등 고정지출을 입력한다.
8. 이미 사용했지만 아직 결제되지 않은 카드 미청구액을 입력한다.
9. 대시보드에서 현재 총 현금과 다음 카드 결제 후 예상 잔고를 확인한다.

### 6.2 일별 지출 입력 시나리오

1. 사용자가 오늘 사용한 지출을 입력한다.
2. 결제수단을 선택한다.
3. 서비스는 결제수단 유형을 확인한다.
4. 신용카드 사용이면 해당 카드의 결제기간에 사용액을 반영한다.
5. 체크카드나 계좌이체 사용이면 연결 통장의 잔고 예측에 즉시 반영한다.
6. 대시보드의 카드 결제예정액과 미래 잔고가 갱신된다.

### 6.3 카드 결제일 예측 시나리오

1. 사용자가 대시보드에 진입한다.
2. 서비스는 카드별 결제일과 결제기간을 기준으로 결제예정액을 계산한다.
3. 해당 결제금액이 연결된 통장에서 출금되는 것으로 처리한다.
4. 고정수입과 고정지출도 함께 반영한다.
5. 사용자는 카드 결제 후 총 현금과 통장별 예상 잔고를 확인한다.

## 8. 화면 기획

### 8.1 온보딩 화면

화면 목적: 사용자가 서비스 사용을 위한 기본 재무 정보를 입력하도록 유도한다.

단계:

1. 통장 등록
2. 카드 등록
3. 카드 결제일 및 결제기간 입력
4. 카드와 통장 연결
5. 고정수입 입력
6. 고정지출 입력
7. 초기 잔고 및 카드 미청구액 입력
8. 대시보드 확인

주요 정책:

- 모든 정보를 한 번에 강제 입력하지 않는다.
- 필수 입력과 선택 입력을 구분한다.
- 카드 결제기간을 모르는 사용자를 위해 안내 문구를 제공한다.
- 초기 설정을 완료하지 않아도 임시 저장 가능해야 한다.

### 8.2 홈 대시보드

화면 목적: 사용자가 현재 현금 상태와 미래 출금 이벤트를 한눈에 파악할 수 있게 한다.

주요 표시 정보:

| 영역 | 내용 |
|---|---|
| 현재 총 현금 | 등록된 모든 통장의 현재 잔고 합계 |
| 다음 카드 결제 | 가장 가까운 카드 결제일과 결제예정액 |
| 결제 후 예상 현금 | 다음 카드 결제 후 남을 것으로 예상되는 총 현금 |
| 다가오는 수입 | 월급, 부업수입 등 예정 입금 |
| 다가오는 지출 | 월세, 보험료, 카드 결제 등 예정 출금 |
| 현금흐름 타임라인 | 날짜별 수입/지출 적용 후 잔고 변화 |

### 8.3 지출 입력 화면

입력 필드:

| 필드 | 필수 여부 | 설명 |
|---|---|---|
| 날짜 | 필수 | 기본값은 오늘 |
| 금액 | 필수 | 지출 금액 |
| 결제수단 | 필수 | 신용카드, 체크카드, 계좌이체, 현금 |
| 사용처 | 선택 | 예: 쿠팡, 스타벅스, 편의점 |
| 카테고리 | 선택 | 식비, 교통, 쇼핑, 고정비 등 |
| 메모 | 선택 | 사용자가 자유롭게 입력 |

주요 정책:

- 신용카드 지출은 통장 잔고에서 즉시 차감하지 않는다.
- 체크카드 지출은 연결 통장 잔고 예측에 즉시 반영한다.
- 계좌이체 지출은 선택한 통장에서 즉시 차감한다.
- 거래 수정 또는 삭제 시 모든 예측값을 재계산한다.

### 8.4 카드 관리 화면

카드 상세 표시 정보:

- 카드명: 현대카드 M
- 카드 유형: 신용카드
- 결제일: 매월 25일
- 결제기간: 전월 13일 ~ 당월 12일

## 11. 핵심 계산 정책

### 11.1 현재 총 현금 계산

현재 총 현금 = 등록된 활성 통장의 현재 잔고 합계

단, 신용카드 사용액은 아직 통장에서 빠져나가지 않았으므로 현재 총 현금에서 즉시 차감하지 않는다.

### 11.2 신용카드 결제예정액 계산

카드 결제예정액 =
해당 결제기간 내 신용카드 지출 합계
+ 해당 결제기간 내 신용카드 고정지출 합계
+ 초기 미청구액 중 해당 결제일에 포함되는 금액

### 11.3 체크카드 지출 처리

체크카드 지출 발생 시: 연결 통장 예상 잔고에서 즉시 차감

### 11.4 계좌이체 지출 처리

계좌이체 지출 발생 시: 선택한 통장 예상 잔고에서 즉시 차감

### 11.5 신용카드 지출 처리

1. 지출 발생일에는 통장 잔고를 차감하지 않는다.
2. 해당 카드의 결제기간에 사용액을 누적한다.
3. 카드 결제일에 연결 통장에서 총 결제예정액을 차감한다.

### 11.6 미래 현금흐름 계산

1. 기준일의 통장별 현재 잔고를 불러온다.
2. 예측 기간 내 고정수입 이벤트를 생성한다.
3. 예측 기간 내 통장 자동이체 고정지출 이벤트를 생성한다.
4. 예측 기간 내 체크카드 지출 이벤트를 생성한다.
5. 예측 기간 내 카드 결제 이벤트를 생성한다.
6. 모든 이벤트를 날짜순으로 정렬한다.
7. 날짜별 이벤트를 적용하며 통장별 예상 잔고와 총 현금을 계산한다.

### 11.7 현재 사용량 고정 모드

포함 항목:

- 현재까지 입력된 신용카드 사용액
- 현재까지 입력된 체크카드 사용액
- 등록된 고정수입
- 등록된 고정지출
- 초기 카드 미청구액
- 카드 결제예정액

제외 항목:

- 아직 발생하지 않은 변동 소비
- 예상 생활비
- 평균 소비 기반 미래 지출
- 할부 예정액
- 리볼빙, 선결제, 카드사 조정금액

| 거래량 기준 | 사용자당 거래 5,000건 수준까지 정상 조회 |
| 계산 방식 | 대시보드 진입 시 실시간 계산 또는 캐싱 혼합 가능 |

### 15.3 사용성

| 항목 | 요구사항 |
|---|---|
| 초기 설정 | 10분 이내 완료 가능해야 함 |
| 지출 입력 | 10초 이내 입력 가능해야 함 |
| 모바일 대응 | MVP부터 모바일 화면 우선 고려 |
| 빈 화면 처리 | 데이터가 없을 때 다음 행동을 안내해야 함 |
| 안내 문구 | 카드 결제기간 입력 시 예시 제공 필요 |

## 16. 주요 정책

### 16.1 카드 결제일 정책

- 사용자는 신용카드별 결제일을 직접 입력한다.
- 결제일은 매월 특정 일자로 설정한다.
- MVP에서는 주말, 공휴일에 따른 실제 출금일 변경은 반영하지 않는다.
- 결제일 변경 이력 관리는 MVP 이후로 미룬다.

### 16.2 카드 결제기간 정책

- 사용자는 카드별 결제기간을 직접 입력한다.
- 결제기간은 예를 들어 전월 13일 ~ 당월 12일 형태로 설정한다.
- 거래일이 결제기간에 포함되면 해당 결제일의 카드 결제예정액에 반영한다.
- 카드사별 특수 규칙은 MVP에서 자동 처리하지 않는다.

### 16.3 초기 미청구액 정책

- 서비스 시작일 이전에 이미 사용했으나 아직 결제되지 않은 카드 금액은 사용자가 직접 입력한다.
- 초기 미청구액은 다음 카드 결제일의 결제예정액에 포함한다.
- 사용자가 초기 미청구액을 입력하지 않으면 예측값이 실제와 다를 수 있음을 안내한다.

### 16.4 고정지출 처리 정책

| 결제 방식 | 처리 방식 |
|---|---|
| 통장 자동이체 | 출금일에 통장 잔고 예측에서 차감 |
| 체크카드 | 결제일에 연결 통장 잔고 예측에서 차감 |
| 신용카드 | 카드 사용액에 포함 후 카드 결제일에 통장에서 차감 |

### 16.5 거래 수정/삭제 정책

- 사용자가 거래를 수정하면 기존 계산 결과는 무효화되고 재계산한다.
- 사용자가 거래를 삭제하면 해당 거래가 반영된 카드 결제예정액 또는 통장 예상 잔고를 재계산한다.
- MVP에서는 거래 변경 이력 저장은 필수 범위에서 제외한다.

## 17. 리스크 및 대응 방안

| 리스크 | 영향 | 대응 방안 |
|---|---|---|
| 초기 입력 부담이 큼 | 사용자 이탈 가능 | 단계별 온보딩, 나중에 입력하기 제공 |
| 카드 결제기간 입력이 어려움 | 잘못된 예측 발생 | 예시 제공, 기본 템플릿 제공 |
| 수동 입력 지속률 저하 | 데이터 품질 저하 | 빠른 입력 UI, 최근 결제수단 자동 선택 |
| 예측값 신뢰도 문제 | 서비스 신뢰 하락 | 계산 근거 노출, 카드별 상세 내역 제공 |
| 금융 데이터 민감성 | 보안 우려 | 계좌번호 전체 입력 제외, 데이터 암호화 |
| 카드사별 예외 다양성 | 계산 복잡도 증가 | MVP에서는 사용자 직접 입력 방식 채택 |

- 카드 등록

- 카드 결제일 입력
- 카드 결제기간 입력
- 초기 잔고 입력

## 19. MVP 릴리즈 기준

1. 사용자가 통장과 카드를 등록할 수 있다.
2. 사용자가 카드 결제일과 결제기간을 입력할 수 있다.
3. 사용자가 카드와 통장을 연결할 수 있다.
4. 사용자가 고정수입과 고정지출을 등록할 수 있다.
5. 사용자가 일별 지출을 입력할 수 있다.
6. 신용카드 사용액이 결제일 기준으로 계산된다.
7. 체크카드 사용액이 통장 잔고 예측에 즉시 반영된다.
8. 다음 카드 결제 후 예상 총 현금을 확인할 수 있다.
9. 현금흐름 타임라인에서 예정 수입과 지출을 확인할 수 있다.
10. 주요 계산 결과가 수동 검산과 일치한다.

## 20. 최종 정리

본 MVP의 핵심은 단순한 가계부 구현이 아니다.

서비스의 본질은 다음과 같다.

사용자가 오늘의 소비를 입력했을 때, 그 소비가 다음 카드 결제일과 미래 통장 잔고에 어떤 영향을 주는지 보여주는 것.

따라서 MVP 개발 과정에서 가장 중요한 판단 기준은 기능의 양이 아니라 카드 결제 후 예상 잔고를 정확하고 이해하기 쉽게 보여주는가이다.

초기 버전은 자동 연동 없이 수동 입력 기반으로 구현한다.
자동화보다 먼저 검증해야 할 것은 사용자가 이 문제를 실제로 중요하게 느끼는지, 그리고 카드 결제일 기반 현금흐름 예측 화면에서 충분한 가치를 느끼는지이다.

MVP의 성공 여부는 다음 문장으로 판단할 수 있다.

`사용자가 이 서비스를 보고 "다음 카드값 빠지고 나면 내가 얼마 남는지 이제 알겠다"라고 느끼는가?`
